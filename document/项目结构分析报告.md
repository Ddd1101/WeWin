# WeWin 项目结构分析报告

## 一、项目概述

WeWin 是一个多电商平台店铺数据管理系统，采用前后端分离架构，支持对接淘宝、天猫、京东、拼多多、1688、抖音电商等多个电商平台，主要用于店铺管理、商品管理、订单同步与分析、库存管理等功能。

## 二、技术栈

### 后端技术栈
- **框架**: Django 5.2.11
- **数据库**: SQLite（开发环境）
- **CORS**: django-cors-headers 4.3.1
- **JWT**: PyJWT 2.8.0
- **HTTP 客户端**: requests 2.32.5

### 前端技术栈
- **框架**: Vue 3.5.25
- **路由**: Vue Router 5.0.3
- **状态管理**: Pinia 3.0.4
- **UI 组件库**: Element Plus 2.13.2
- **HTTP 客户端**: Axios 1.13.6
- **图表库**: ECharts 6.0.0
- **构建工具**: Vite 4.5.3

## 四、核心模块详解

### 4.1 后端核心应用

#### 1. account 应用 - 账户管理
- **功能**: 用户认证、权限管理、角色管理
- **模型**: 自定义用户模型（User）
- **管理命令**: 
  - `create_superadmin`: 创建超级管理员
  - `init_page_config`: 初始化页面配置
  - `update_enterprise_admin_to_leader`: 更新企业管理员

#### 2. company 应用 - 企业管理
- **功能**: 企业信息管理、企业与用户关联

#### 3. store 应用 - 店铺/商品/订单管理（核心）
**核心数据模型**：

**商品管理**：
- `Product`: 商品基础表（货号、名称、类型、成本价、售价、图片等）
- `Bead`: 串珠特有属性（材质、尺寸、颜色、重量、品质等级等）
- `Accessory`: 手串配件特有属性
- `FinishedProduct`: 手串成品特有属性（工费、弹性成本）
- `FinishedProductBead`: 成品与串珠的关联表（多对多）
- `FinishedProductAccessory`: 成品与配件的关联表（多对多）

**店铺管理**：
- `Store`: 店铺表（名称、平台、品类、关联企业、管理员等）
- `PlatformApiConfig`: 平台 API 配置（App Key、Secret、Token 等）
- `StoreDataConfig`: 店铺数据拉取配置
- `StoreData`: 店铺数据记录

**订单管理**：
- `Order`: 订单表（核心）
- `OrderItem`: 订单商品项
- `OrderReceiver`: 订单收货人信息
- `OrderContact`: 订单联系人
- `OrderStep`: 订单步骤
- `OrderTradeTerm`: 订单交易条款
- `OrderRateInfo`: 订单评价信息
- `OrderBizInfo`: 订单业务信息
- `OrderLogistics`: 订单物流信息
- `OrderLogisticsItem`: 订单物流明细

**数据同步**：
- `DataPullTask`: 数据拉取任务
- `StoreDataConfig`: 店铺数据配置

### 4.2 前端核心模块

**路由配置**：
- `/login`: 登录
- `/register`: 注册
- `/bind-company`: 绑定企业
- `/dashboard`: 数据概览
- `/stores`: 店铺管理（列表、订单总览）
- `/products`: 商品管理
- `/inventory`: 库存管理
- `/sales`: 销售数据
- `/users`: 用户管理
- `/profile`: 个人信息
- `/companies`: 企业管理（需要管理员权限）

### 4.3 AliData 模块

**功能**: 专门处理 1688 平台的数据处理
- `ali_api.py`: 1688 API 调用
- `OrderManager.py`: 订单管理器
- `cloth_trade_manager.py`: 布料交易管理器
- `cloth_worksheet.py`: 布料工作表处理

## 五、数据库设计要点

### 5.1 商品体系设计
采用**基础商品表 + 类型扩展表**的设计模式：
- `Product` 表存储通用信息
- `Bead`、`Accessory`、`FinishedProduct` 表通过一对一关联扩展特定属性
- 成品通过 `FinishedProductBead` 和 `FinishedProductAccessory` 表与原材料建立多对多关系

### 5.2 订单体系设计
采用**主订单表 + 扩展详情表**的设计模式：
- `Order` 表存储核心订单信息
- 其他表（`OrderItem`、`OrderReceiver`、`OrderLogistics` 等）通过外键关联，按需扩展
- 支持多平台订单存储（通过 `Store` 关联）
- 保留平台原始数据（`platform_raw_data` 字段）

### 5.3 多平台支持
通过 `Store` 表的 `platform` 字段支持多种电商平台：
- 淘宝、天猫、京东、拼多多、抖音电商、快手电商、1688、小红书、微信小店等

## 六、核心业务流程

### 6.1 订单同步流程
1. 配置店铺 API 信息（`PlatformApiConfig`）
2. 创建数据拉取任务（`DataPullTask`）
3. 调用对应平台 API（如 1688）
4. 解析订单数据并保存到数据库
5. 更新数据拉取状态

### 6.2 商品管理流程
1. 创建商品基础信息（`Product`）
2. 根据商品类型创建对应扩展记录（`Bead`/`Accessory`/`FinishedProduct`）
3. 对于成品，配置其组成的串珠和配件
4. 管理商品图片、库存等信息

## 七、项目启动与部署

### 7.1 后端启动
```bash
cd Server
# 激活虚拟环境
.\venv\Scripts\Activate.ps1
# 运行开发服务器
python manage.py runserver
```

### 7.2 前端启动
```bash
cd Page
# 安装依赖
npm install
# 开发模式
npm run dev
# 构建生产版本
npm run build
```

### 7.3 部署相关
- 后端可使用 Gunicorn + Nginx 部署
- 前端构建后使用静态文件服务部署
- 生产环境建议使用 PostgreSQL/MySQL 替代 SQLite

## 八、项目特色

1. **多电商平台支持**: 统一管理多个电商平台的店铺和订单
2. **灵活的商品体系**: 支持串珠、配件、成品等多种商品类型，以及成品的组合管理
3. **完整的订单数据**: 保存订单的完整信息，包括买家、卖家、商品、物流、评价等
4. **平台 API 集成**: 内置 1688 等平台的 API 对接能力
5. **前后端分离**: 现代化的技术架构，便于维护和扩展

## 九、后续优化建议

1. **数据库迁移**: 生产环境建议升级到 PostgreSQL 或 MySQL
2. **缓存机制**: 引入 Redis 缓存，提高热点数据访问速度
3. **异步任务**: 使用 Celery 处理订单同步等耗时任务
4. **单元测试**: 完善核心业务的测试覆盖
5. **API 文档**: 使用 Swagger/OpenAPI 完善 API 文档
